# HealthView OS Data Sources

HealthView OS should gather health information from many places while
minimizing access, preserving provenance, and asking the user only when
permission, ambiguity, or risk requires it.

This document describes how HealthView OS should acquire data. It should be
read with `PRIVACY.md`, `DATA.md`, and `MODEL.md`.

## Core Rule

> First use available authorized local data, then use task-scoped acquisition
> methods, and ask the user only when blocked by permission, authentication,
> ambiguity, or consequential external action.

HealthView OS must not collect everything merely because access is available.

## Origin And Acquisition

HealthView OS must distinguish where information came from and how it was
obtained.

```text
Origin: Northside Medical Group
Acquisition: SMART on FHIR API
```

```text
Origin: Northside Medical Group
Acquisition: PDF imported from user-selected folder
```

```text
Origin: Community Laboratory
Acquisition: Lab portal download through user-authorized browser session
```

The origin is the real-world source of the health information. The acquisition
method is the path HealthView OS used to get it.

## Source Trust And Freshness

Evidence should carry source quality metadata so the UI can explain why a claim
is shown and how much weight it deserves.

Recommended trust levels:

- `user_entered`: entered directly by the user.
- `imported_document`: imported document, image, PDF, CSV, or export.
- `provider_api`: provider or EHR API.
- `signed_clinical_document`: signed or structured clinical document.
- `lab_result`: laboratory result or diagnostic report.
- `payer_claim`: payer claim, explanation of benefits, authorization, or bill.
- `device_or_wearable`: device, wearable, or health platform reading.
- `ai_derived`: generated by HealthView OS or an AI processor.
- `public_directory`: public provider, facility, pricing, or service data.

Recommended freshness values:

- `current`
- `stale`
- `superseded`
- `unknown`

The same fact from a signed lab report and a user note should not carry the
same evidence weight.

## Acquisition Methods

### User Provided

Information entered directly by the user.

Examples:

- Forms.
- Notes.
- Corrections.
- Symptoms.
- Goals.
- Manual measurements.
- Chat messages explicitly saved by the user.

### User Resources

Information stored in places the user controls or grants access to.

Examples:

- Local files.
- Imported PDFs, images, CSVs, and exports.
- Selected cloud files or folders.
- Email attachments.
- Wearable exports.
- App exports.

HealthView OS should prefer explicit file or folder selection over broad
account access.

### Native Health Platforms

Information accessed through device-level health platforms after user
permission.

Examples:

- Apple HealthKit.
- Android Health Connect.
- Wearable and fitness app APIs.

These sources are likely more relevant once HealthView OS has native mobile or
desktop adapters.

### Provider And Payer APIs

Information accessed through supported healthcare APIs.

Examples:

- SMART on FHIR provider connections.
- Payer Patient Access APIs.
- Medicare Blue Button.
- Pharmacy, lab, imaging, or benefits APIs where available.

API access should use the narrowest practical scopes for the user's task.
Adapters should pin the exact FHIR, US Core, payer, or vendor version they map.

### Provider And Payer Portals

Information accessed through portals when APIs are unavailable, incomplete, or
not practical.

Examples:

- Provider patient portals.
- Insurance portals.
- Lab portals.
- Pharmacy portals.
- Imaging center portals.

HealthView OS may use browser-based assistance to navigate portals, find
records, download documents, and import relevant files. Browser use must stay
task-scoped and user-visible when practical.

### Record Requests

Information requested from a provider, payer, lab, imaging center, pharmacy, or
other data holder when direct access is unavailable.

HealthView OS may help the user prepare, submit, and track record requests.

### Public Data

Non-private data used to support Services, Billing, and care navigation.

Examples:

- Provider identifiers.
- Provider locations and specialties.
- Facility directories.
- Public price transparency files.
- Public plan or service information.

Public data can be handled differently from personal health data, but its
source and update time should still be recorded.

## Agent Retrieval Policy

When the user asks a question or requests a task, the agent should follow this
order:

1. Search existing local HealthView OS data.
2. Determine what is missing, stale, or insufficient for the task.
3. Use already-authorized sources relevant to the task.
4. Try reasonable autonomous acquisition paths:
   - Supported APIs.
   - Native health platforms.
   - Selected files, folders, cloud storage, or email.
   - Browser-based portal use in the user's authorized session.
5. Retrieve only the minimum necessary data.
6. Record origin, acquisition method, permissions, timestamps, source artifact,
   freshness, and trust level.
7. Ask the user only when blocked.
8. Fall back to record requests or manual import when other reasonable methods
   fail.

## When To Ask The User

The agent must ask before or when it needs:

- New permissions.
- Login, MFA, or CAPTCHA.
- Consent.
- A choice between ambiguous patients, accounts, files, portals, or records.
- Access broader than the current task requires.
- Transmission of identifiable health information to an external party.
- A consequential external action such as booking, canceling, submitting,
  sending, requesting, paying, or changing permissions.

The agent should not ask the user to manually download and upload files until
reasonable autonomous options have failed or are not appropriate.

## Browser-Based Acquisition

Browser-based acquisition is valid when APIs are unavailable or insufficient.

Preferred model:

```text
User authorizes or logs in
-> HealthView OS assists in the user-controlled browser session
-> HealthView OS finds relevant records or downloads
-> User confirms consequential actions
-> HealthView OS imports selected data
-> HealthView OS records origin, acquisition method, and evidence
```

Rules:

- Keep browser use constrained to the user's requested task.
- Avoid broad portal exploration.
- Avoid background credential use.
- Do not store portal credentials unless a reviewed credential strategy exists.
- Confirm before downloads that expose files outside the vault when required by
  browser or product policy.
- Confirm before sending messages, submitting forms, booking, canceling,
  paying, requesting records, or changing permissions.

## UI Evidence Policy

Important UI claims should be explainable.

The UI should be able to expose:

```text
Claim
Evidence
Confidence
Source freshness
Generated by
Last updated
User action
```

Example:

```text
Claim: Inflammation markers rising
Evidence: CRP values from Quest lab reports on Jan 8, Mar 2, and Jun 5
Confidence: Medium
Freshness: Current
Generated by: HealthView OS trend rule v0.1
Action: Review sources
```

Not every small UI label needs a drawer, but every warning, score,
recommendation, or health-system status should have an evidence path.

## Provenance

Every acquired item must record:

- Origin.
- Acquisition method.
- Import or sync time.
- Permission or authorization context, where applicable.
- Source trust level.
- Source freshness.
- Original source artifact.
- Transformations performed after acquisition.
- Any uncertainty about the source or interpretation.

This provenance feeds the `DATA.md` and `MODEL.md` policies.

## Out Of Scope For Now

- AI source memory.
- Shared provider or payer capability catalogs.
- Long-term credential management.
- Background portal automation.
- Cross-user learning from acquisition attempts.

These should be designed as part of the broader AI memory, permissions, and
security strategy before implementation.
